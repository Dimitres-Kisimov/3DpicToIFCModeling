## 1. Objective

Turn a **photo of furniture** into a **BIM-compliant IFC** model, and scale that to **auto-furnishing
whole buildings**: load a real architectural IFC, read its rooms, populate each ergonomically, view in
xeokit, export IFC/GLB. Commercial framing: SCS sells **outputs** (IFC + configured rooms), not model
weights → only MIT/Apache/SAM/LGPL/CC-BY components are used (royalties = $0).

## 4. The building-population system (centerpiece — how it works)

**Entry:** `localhost:8000` → Building dropdown → **Duplex Apartment** → per-room chips → **Populate**.

**Flow:**
```
sample_buildings/Duplex_Architecture.ifc  (real: 21 rooms, walls, doors, 4 storeys, IFC2X3, 2.4 MB)
   │  backend/python-scripts/populate_building.py   (called by app_server.py)
   ▼
 read IfcSpace rooms (name+footprint)  →  smart_furnish(type,area)  →  extract obstacles
 (internal/party walls, beams, columns, stairs, doors) → MERGE overlapping obstacles (shapely) →
 CP-SAT solver (spatial_layout.py + rule_packs.py: Neufert/Panero/ADA) places AROUND obstacles →
 fit-as-many-as-possible (shrink set until all placed cleanly) → 0 clashes
   │  --movable mode → shell.glb + one GLB per piece + furniture.json (each piece world pos)
   ▼
 app_server.py /populate returns {shell, pieces[]}; frontend loads them as SEPARATE xeokit models →
 click-pick + drag on floor (surface pick → worldPos, reload-on-release) → /save merges dragged
 positions + shell → downloadable building.glb
```

**Verified on Duplex:** ~5–8 rooms furnished, ~18–23 pieces, **0 clashes**. Living→sofa/table/lamp/
shelf; Bedroom→bed/wardrobe/(+study if big); Kitchen→cabinets/table; bathrooms/halls/stairs skipped.

**Backend endpoints (Flask, `backend/app_server.py`):**
- `GET /api/buildings` · `GET /api/building/<id>/rooms` (rooms+type+area+suggestion) ·
  `POST /api/building/<id>/populate` (→ shell+pieces) · `POST /api/building/<id>/save` (→ merged GLB)
- `POST /api/upload_generated` (drop .glb/.ifc → auto-categorize → `data/generated_assets/manifest.json`)
  · `GET /api/generated/<fn>` · plus existing `/api/catalog`, `/api/items/<cat>`, `/api/generate`.

**Frontend:** `demo/app.html` + `demo/app.js` (loaded `?v=14`). Building panel + per-room chips +
drag logic (`bPieces`, `loadBuilding`, pointer pick/drag) + Save + drop-zone/upload + "OURS" badge.

## 7. Technical notes for continuity

- **Coordinates:** IFC is **Z-up** (floor=XY, Z vertical); glTF/xeokit is **Y-up**. Convert with rotation
  −90° about X: `(x,y,z)→(x,z,−y)`. Asset GLBs are Z-up + real-scaled, so furniture drops in with a
  yaw-only rotation; the whole scene is rotated to Y-up at the end for the viewer.
- **Rooms are treated as bounding boxes** (obstacles handle intrusions). Real polygon clipping is a TODO.
- **XKT is a JSON fallback** (the `@xeokit/xeokit-convert` CLI isn't installed) — loads, but not binary/fast.
- **Metric F-score** primary eval: `F = 2PR/(P+R)`. Benchmark mean: TripoSG 0.393 > SAM3D 0.368 >
  TRELLIS 0.347 > InstantMesh 0.327 > TripoSR. Asset library = best-of-each.
- **Licenses/royalties = $0** for the shipped stack (MIT/Apache/SAM/LGPL/CC-BY). Traps to keep out:
  Depth-Anything **Large** (CC-BY-NC = non-commercial; only *Small* is Apache), StableFast3D (Stability
  community, $1M cap), Hunyuan3D (Tencent). Compute ~$37–85/mo pilot → <$0.005/room at scale.

## 8. ⚠️ Known issues / gotchas

- **Live Generate is slow (~2 min) and 6 GB-VRAM-bound.** It segfaults (`0xC0000005`/exit 139) if the GPU
  state gets poisoned by heavy stacked runs — **reboot fixes it**. Generate ONE at a time; close heavy
  3D tabs if it OOMs. All CPU building/IFC tools are unaffected.
- **Servers die on session cycle** unless launched **detached** (PowerShell Start-Process — see §3).
- **spatial_layout rotated-centre bug** — corrected in populate_building, not at source (shared with room app).
- **Not one app yet** — see §9.
