📁 Code files and what they do

every button, gesture and file — and how the UI maps to the engines underneath

← Research hub
How the app is built, in one paragraph: a Node/Express server (backend/server.js) serves the one-page UI (frontend/) and a REST API (backend/routes/). Every heavy operation — mesh generation, catalog work, the CP-SAT layout solver, building population, IFC export — runs in Python (backend/python-scripts/), spawned per request through services/roomApi.js with queues so bursts can't pile up. One shared 3D viewport (xeokit) hosts three workspaces; switching tabs never destroys work. The German ASR rulebook lives in ONE file (rule_packs.py) that both the room and the building engines read — change a rule once, both obey. The classic clickable tree view lives here.

6 · One click, end to end how the UI relates to functionality

  • Generate: photo → objectRoutes → rembg segmentation → TripoSR mesh → repair pack + chair-base graft → real dimensions (Neufert) → thumbnail → auto-register with a professional code → appears in the viewer + the catalog.
  • Suggest & solve: type/size/density → smart_furnish (ASR staffing, capacity guard) → your edits and ⋯ picks → build_room_scene pre-passes (rows, safety strips, armchair rows) → CP-SAT → children attach → wall decor mounts → GLB + 2D plan + the refusal list.
  • Building: drop an IFC → profile + registry → geometry scan → per-room cards → picks/blueprints → populate_building per room (same rulebook) → movable pieces in 3D/2D → drag/rotate → 💾 GLB / 📥 IFC with everything baked in.
