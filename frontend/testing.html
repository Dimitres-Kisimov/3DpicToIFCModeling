T4 · Malformed-IFC fuzzing 2026-07-09 PASS — after truncation fix

Five hostile uploads against the registry. Acceptance criteria: a clean, actionable refusal for each; no stuck "preparing" entries; server healthy afterwards.

CaseFirst runAfter fix
Truncated mid-entity (first 100 KB of a real 9.7 MB IFC) ACCEPTED — registered & prepared as a seemingly valid building refused: "file is incomplete (missing STEP terminator)"
Plain text renamed .ifcrefused: "not an IFC (STEP) file"same
Valid ISO-10303 header, garbage entitiesrefused (parse error)same
Schema-valid IFC4 with zero IfcSpacerefused with re-export guidancesame
Wrong extension (.glb)refused by the extension gatesame

The bug this found: ifcopenshell happily salvages the parseable prefix of a truncated file — the 100 KB stub of the German apartment contained the project header and a few early rooms, so it sailed through the probe and registered as a plausible-looking building that silently lacked 99% of its content. The upload gate now also reads the file tail and requires the STEP terminator END-ISO-10303-21, so a cut-off upload or export is named as exactly that. Re-run: 5/5 refused, registry unchanged, 0 stuck states, server healthy.
